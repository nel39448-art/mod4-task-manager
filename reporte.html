1. Introducción

En la clase correspondiente a la Sesión 3 del Módulo 4, avanzamos en el tema de las pruebas de software para APIs construidas con Express. Vimos cómo usar Vitest y la librería Supertest para simular peticiones HTTP sin tener que levantar el servidor localmente en cada prueba.

Para poner esto en práctica, realizamos el Laboratorio 3 llamado "Cacería de Bugs". Este ejercicio consistía en recibir un código con un bug en un endpoint de creación de tareas, escribir una prueba que demuestre que el error existe y después corregirlo. En este informe explico paso a paso lo que hice en el código y cómo logré superar las pruebas, incluyendo los puntos del reto adicional.

2. Ronda 1: Escribir la prueba y reproducir el error

Primero instalé las dependencias del proyecto (express, vitest y supertest) y preparé los archivos base. Escribí una primera prueba en el archivo test/app.test.js. Esta prueba hace una petición POST al endpoint /tareas con un título específico y espera que el servidor responda con un estado 201 y devuelva los datos de la tarea creada.

Al correr el comando npm run test en la terminal, la prueba falló. Esto ocurrió porque el servidor estaba devolviendo un estado 200 en lugar del 201 que se esperaba, y además respondía con un mensaje estático que no contenía el título de la tarea. En las siguientes imágenes se puede ver el error en la terminal y la parte del código de src/app.js que tenía el bug.

Imagen 1: Terminal de VS Code mostrando que la prueba falló (se esperaba 201 pero llegó 200).
Imagen 2: Código original de app.js mostrando la respuesta incorrecta del servidor en la línea 18.

3. Ronda 2 y 3: Corregir el bug y verificación básica

Para solucionar el bug, abrí el archivo src/app.js y modifiqué la respuesta del endpoint POST. Cambié la línea de respuesta para que use el código de estado 201 y envíe el objeto de la tarea creada (que llamamos nuevaTarea) en formato JSON.

Después de guardar el archivo, ejecuté de nuevo el comando de pruebas en la terminal de VS Code. Como se muestra en la Imagen 3, la prueba pasó satisfactoriamente y la terminal se puso en verde, indicando que el primer bug fue resuelto correctamente.

Imagen 3: Servidor corregido y terminal mostrando la prueba aprobada.

4. Reto extra: Validación y pruebas adicionales (Triple Verde)

Para completar los requisitos del reto adicional y ganar los 15 puntos extras de la insignia "Triple Verde", agregué las siguientes dos funcionalidades al código:

  • Una ruta GET /tareas para listar todas las tareas guardadas en la memoria.
  • Una validación en el POST para revisar que el título de la tarea no llegue vacío. Si llega vacío, el servidor ahora responde con un código de error 400 (Bad Request).

Escribí las pruebas correspondientes para estos dos casos en el archivo de pruebas y luego implementé el código necesario en el servidor. Al correr nuevamente la suite de pruebas completa, las tres pruebas pasaron al mismo tiempo sin problemas, como se ve en la Imagen 4.
